# Shyraq BP

## **1. Kindergarten Enrollment and Onboarding**

## Process

Зачисление ребенка в садик

### Goal

Провести путь от заявки родителя до создания карточки ребенка и подготовки первого начисления.

### Actors

- Родитель
- Администратор садика
- Система

### Trigger

Родитель оставляет заявку на поступление ребенка в садик.

### Platforms

- Admin Web
    - Lead Statuses in Onboarding :
        - `Новая заявка` - заявка зарегистрирована, но еще не обработана администратором.
        - `В обработке` - администратор проверяет заявку, собирает данные и оценивает возможность зачисления.
        - `Лист ожидания` - подходящего места сейчас нет, но заявка остается активной для последующего возврата в работу.
        - `Создана карточка` - карточка ребенка создана, ожидает первой оплаты и первого посещения ребенка.
        - `Отменен` - зачисление прекращено без создания карточки ребенка.
        - `Архив` - карточка ребенка переведена в архив после прекращения активного периода *(перестал ходить в дет. сад)*
- Система

### Main Flow

1. Родитель оставляет заявку на поступление ребенка.
2. Администратор открывает карточку нового лида (`Новая заявка`) на зачисление в `Admin Web`.
3. При редактировании карточки лида статус: `В обработке`
    1. Администратор: вносит персональные данные ребенка (заполняет карточку)
        1. ФИО ребенка*
        2. Возраст ребенка*
        3. Основной законный представитель* *(Все следующие связи, добавление родителя, няни идут только через `Approval` от основного законного представителя.)*
            1. ФИО представителя*
            2. Номер Телефона* *(Проверяется по OTP при регистрации)*
            3. Связь с ребенком (Кем является ребенку)*
    2. Администратор выбирает группу для ребенка
4. После заполнения данных и выбора группы система переводит статус в `Создана карточка`.
5. Система создает обязательство на первый месяц оплаты.
6. Процесс передает ребенка в процесс `Payments`.

### Alternative Flows

- Если свободного места нет, администратор переводит заявку в лист ожидания.
- Позже родитель получает информацию о доступности места.

### Result

- Заявка в статусе: `Создана карточка`
    - Ожидает Оплаты
    - Ожидает Первый день ребенка в садике
- Заявка в статусе: `Лист ожидания`
    - Нет подходящего места в садике (К примеру: нет свободных мест в группе)

### Дополнения (B4 manual creation)

- Параллельно с enrollment-машиной admin может создать карточку ребёнка вручную через `POST /admin/children` (статус `card_created`, без `enrollments`-trail, без авто-invoice'а на первый месяц). Этот путь нужен для случаев, когда ребёнок мигрировал из другого садика или enrollment-процесс был оффлайн (бумажный договор, телефонный разговор).
- Различие с enrollment-flow: при manual creation нет лида и `enrollment_status_log`; ребёнок сразу получает `children.status='card_created'`, но без обязательства на оплату — admin отдельно назначает `tariff_assignment` (B13). Все остальные правила (валидация ИИН, привязка к группе, добавление primary guardian с approval-flow) — те же.

## 2. Parent App Onboarding

## Process

Регистрация родителя и привязка ребенка к аккаунту

### Goal

Дать родителю доступ к кабинету ребенка через `Parent App`.

### Actors

- Родитель/Няня
- Система

### Trigger

Родитель/Няня устанавливает и открывает `Shyraq Parent App`.

### Platforms

- Parent App
- Admin Web

### Main Flow

1. Родитель/Няня регистрируется по номеру телефона в `Parent App`.
2. Система отправляет SMS-код подтверждения.
3. Родитель подтверждает номер телефона кодом.
4. Система создает или активирует учетную запись родителя.
5. Родитель/Няня выбирает действие `Добавить ребенка`.
6. Родитель/Няня вводит ИИН ребенка.
7. Система ищет ребенка, ранее созданного в `Admin Web`.
8. Если ребенок найден и данные совпадают, система предлагает подтвердить привязку.
9. Родитель/Няня подтверждает связь.
10. Ожидается `Approval` в приложении от `Primary Guardian` 
    1. `Primary Guardian` выбирает право родителя/няни для `Approval` действий *(максимум 2 родителя)*
11. Система связывает аккаунт родителя с карточкой ребенка.
12. Родитель/Няня получает доступ к кабинету ребенка.
13. Дальше процесс продолжается в `Notifications and Parent Visibility`.

### Alternative Flows

- Если `ребенок не найден`, система сообщает, что ребенок еще не создан или данные не совпали.
    - Родитель обращается в садик/поддержку.
    - Администратор проверяет наличие карточки ребенка и корректность ИИН.
    - После исправления родитель повторяет привязку.
- Если родитель регистрируясь является `Primary Guardian` *(заполняется в **Enrollment and Onboarding**)*
    - Ребенок уже привязан к аккаунту родителя *(через номер телефона)*
    - Родитель может воспользоваться действием `Добавить ребенка`

### Result

- Родитель связан с карточкой ребенка и получает доступ к данным.
    - С правом `Approval`
    - Без права `Approval`

## 3. Staff and Admin Account Provisioning

## Process

Создание и активация учетной записи сотрудника

### Goal

Подготовить сотрудника к работе в нужном контуре и с нужными правами.

### Actors

- Руководитель садика
- Администратор
- Сотрудник
- Система

### Trigger

Руководитель принимает нового сотрудника.

### Platforms

- Admin Web
- Staff App

### Main Flow

1. Администратор открывает раздел сотрудников в `Admin Web`.
2. Администратор создает нового сотрудника и заполняет нужные поля.
    1. ФИО
    2. Телефон
    3. Роль
        1. Если это `Mentor`, администратор назначает одну или несколько групп.
        2. Если это `Specialist`, администратор назначает тип специалиста.
            1. Каждый специалист имеет свои Формы заполнения при работе
        3. Если это `Admin`, администратор назначает доступ к заявкам, детям, контенту и сводной статистике.
3. Система активирует учетную запись сотрудника.
4. Сотрудник входит в `Staff App` по номеру телефона.
5. Система отправляет OTP.
6. Сотрудник подтверждает вход кодом.
7. Если у сотрудника одна роль, система открывает соответствующий домашний экран.
8. Если у сотрудника несколько ролей, система показывает выбор роли.
9. Сотрудник выбирает нужную роль.
10. Система открывает рабочее пространство выбранной роли.

### Result

- Сотрудник активирован и может работать в `Staff App` с корректными правами доступа.

### Дополнения (архитектурные)

- **Правила mentor ↔ group:**
  1. **Один mentor = ровно одна активная группа.** Enforced: `idx_group_mentors_one_active ON group_mentors(staff_member_id) WHERE unassigned_at IS NULL`. При переводе — старое назначение закрывается (`unassigned_at=NOW()`), создаётся новое.
  2. **У группы может быть несколько менторов**, но **ровно один `is_primary=true`** (основной). Enforced: `idx_group_mentors_one_primary_per_group ON group_mentors(group_id) WHERE is_primary=TRUE AND unassigned_at IS NULL`.
  3. Первый назначаемый mentor группы автоматически становится primary. Смена primary: сначала снять флаг у текущего (в той же TX), потом назначить новому.
- **Specialist может обслуживать несколько групп/детей** — для специалиста ограничения нет.
- **Admin и Reception** — не привязаны к группе.
- **Identity QR** выдаётся каждому staff-пользователю (для авторизации у турникета, если Face ID не зарегистрирован).
- **Атомарность create-staff + assign-mentor (B3).** `POST /admin/staff` с `role=mentor` и переданным `group_id` выполняет вставку `staff_members` и `group_mentors` в одной транзакции. Если у группы ещё нет активного primary — новый mentor становится primary (`is_primary=true`) автоматически. Инварианты `idx_group_mentors_one_active` / `idx_group_mentors_one_primary_per_group` enforced на DB-уровне (partial unique indexes); гонка → `P2002` → 409 `mentor_one_active_group_violation` / `group_primary_conflict`.
- **Bootstrap первого admin'а (B2 Tenant Bootstrap).** Шаги 1–2 основного flow выполняет уже существующий admin — но **первый** admin садика создаётся иначе: SuperAdmin через `POST /super-admin/kindergartens` в одной транзакции создаёт `kindergartens` + `users` (find-or-create по phone) + `staff_members(role=admin, is_active=true)`. Активация admin'а идёт по welcome-SMS («Кабинет <name> готов, войдите по телефону <phone>») + обычный OTP-flow §0.1 `/auth/otp/request`+`/auth/otp/verify` — **отдельного invite-token / magic-link нет**. Начиная со второго staff-члена — основной flow §3 (admin создаёт в Admin Web).

## 4. Payments

## Process

Оплата основного и дополнительного начисления

### Goal

Обеспечить прозрачный цикл начисления и оплаты для родителей и администрации.

### Actors

- Родитель
- Администратор
- Платежная система
- Система

### Trigger

У ребенка появляется начисление на оплату. 

- Ежемесячная оплата
- Оплата долга за услугу

### Platforms

- Parent App
- Admin Web

### Main Flow

1. После создания ребенка система формирует обязательство на первый месяц.
2. Администратор видит начисление в карточке ребенка в `Admin Web`.
3. Родитель после привязки ребенка видит статус месяца `Не оплачено` в `Parent App`.
4. Родитель открывает страницу оплаты.
5. Система показывает сумму, период и доступный способ оплаты.
6. Родитель выбирает способ оплаты.
    1. Досрочная оплата
        1. 3 месяца
        2. 6 месяцев
        3. 12 месяцев *(цена со скидкой)*
    2. Частичная оплата
        1. Ввод суммы за этот месяц
7. Платежная система подтверждает оплату.
8. Система меняет статус месяца на `Оплачено`.
9. Родитель видит обновленный статус оплаты.
10. Администратор видит подтвержденную оплату.

### Payment Options

- **`Halyk ePay`** — Halyk Bank, любые карты
- **`Kaspi Pay API`** — Kaspi.kz (не статичный QR, а нативный API)
- **`TipTopPay`** ИЛИ **`Freedom Pay`** — для карт других банков (финальный выбор провайдера — до интеграции)

### Alternative Flows

### Новый расчетный период

1. Наступает новый расчетный период.
2. Система создает новое начисление за следующий месяц.
3. Родитель видит новый месяц со статусом оплаты.
4. Родитель повторяет процесс оплаты.

### Дополнительная платная услуга

1. Система или администратор создает дополнительное начисление.
2. Родитель видит отдельную задолженность или доплату.
3. Родитель оплачивает ее.
4. Система обновляет платежный статус.

### Result

- Начисление оплачено или ожидает оплаты.

### Дополнения (архитектурные)

- **Фискализация (ОФД РК):** на каждую успешную оплату (`payment.status = completed`) система автоматически формирует фискальный чек через интегрированного ОФД-провайдера (Kassa24 / Rekassa / Webkassa). Родитель получает `qr_url` для проверки. Admin Web видит очередь выданных и failed чеков.
- **Тарифная сетка (гибкая):** цена задаётся через `tariff_plans` (базовая месячная, доп.услуги, поздний забор, апгрейды питания) с `discount_rules` (скидки за братьев/сестёр, prepay 3/6/12 месяцев, льготные категории). Каждому ребёнку назначается `tariff_assignment` на период.
- **Pro-rata:** если ребёнок архивируется посреди месяца — cron пересчитывает оставшийся период, учитывая `kindergarten_holidays.is_billable = false`, и создаёт `refund` либо negative line item в следующий invoice.
- **Возвраты:** отдельная сущность `refunds` со статусом (pending/processed/failed), связью с payment и invoice, причиной и провайдер-референсом для платёжной системы. **Статус-машина возвратов:** `pending → {approved → processed, rejected}`.
- **Ежемесячная генерация:** cron `billing:invoice-generate` 1-го числа месяца (02:00 Asia/Almaty, BullMQ repeatable) проходит по активным `tariff_assignments` и создаёт `invoices` + `invoice_line_items`. Ручной триггер для super-admin: `POST /saas/billing/monthly-run`.
- **Custom скидки (новая подсистема):** админ настраивает скидку с условиями и таргетингом — см. подраздел `4.1 Custom Discounts` ниже.
- **Статус-машина инвойсов:** `pending → {partial, paid, overdue, refunded, cancelled}`. `partial` — когда сумма поступивших платежей > 0 но < `amount_after_discount`; `paid` — полное покрытие; `overdue` — cron переводит при `now > due_date AND status IN ('pending','partial')`; `refunded` — после обработки полного возврата; `cancelled` — admin или система.
- **Статус-машина платежей** (`payments` таблица): `initiated → {processing, completed, failed, refunded}`. Переход `completed` — через webhook от провайдера; `failed` — при ошибке провайдера или ручном admin-cancel; `refunded` — после обработки `refunds`.
- **Провайдер-агностичность (B13):** платёжный провайдер скрыт за `PaymentProviderPort` (абстрактный класс). В B13 реализован `MockPaymentProvider` (детерминированный, без внешних HTTP-вызовов). Реальные провайдеры (Halyk ePay, TipTopPay, FreedomPay) подключаются в Phase B без изменений бизнес-логики (`PAYMENT_PROVIDER=mock|halyk|tiptoppay|freedompay`, default `mock`). Аналогично `FiscalReceiptPort` + `MockFiscalReceiptAdapter` (B13); реальный OFD — B15.
- **Хук enrollment → первый инвойс:** при переходе enrollment в `card_created` (BP §1) система автоматически вызывает `BillingService.generateFirstInvoice(kgId, childId, enrollmentDate)` в ambient TX. Закрывает `TODO(B13)` в `EnrollmentService.transition()`.
- **Хук late_pickup → инвойс:** при `accept` заявки типа `late_pickup` (BP §6) система вызывает `BillingService.generateLatePickupInvoice(kgId, childId, parentRequestId)` и проставляет `parent_requests.invoice_id`. Закрывает `TODO(B13)` в `ParentRequestService.accept()`.
- **B13 demo-ready (Phase A):** функционал §4 (без §4.1) закрыт на Mock-адаптерах в коммите `df1c9af`. Тарифы, invoice CRUD, monthly cron, parent pay (full / partial / prepayment 3/6/12/24 м), webhook flow, refund flow (pending → approved → processed), pro-rata holidays. Provider-agnostic ports готовы — реальный Halyk ePay подключается одной строкой через `PAYMENT_PROVIDER=halyk` в B14 / Phase B. Реальный ОФД — через `FISCAL_PROVIDER=<vendor>` в B15.

### 4.1 Custom Discounts (праздничные и спец. скидки)

**Demo-ready (B16).** Conditions engine, stacking, daily-expire cron, admin CRUD + state transitions, InvoiceService integration — закрыто на Mock-адаптерах в коммитах `53f17c4` → `8c2b094`.

**Цель:** админ может настраивать скидки (праздники, льготы, промо) с условиями, таргетингом и автоматическим push-уведомлением родителям. Скидки вступают в силу сразу.

**Actors:** Admin, Система, Родитель.

**Поддерживаемые условия** (в `custom_discounts.conditions` JSONB):

| Условие | Пример применения |
|---|---|
| **Предоплата N месяцев** | Оплата на 24 месяца вперёд — скидка 15% |
| **Количество детей в семье** | 3+ детей → скидка 20% |
| **Возраст ребёнка** | Младшая группа (1–3 года) — скидка 10% |
| **Категория льгот** | Многодетная семья, инвалидность, мать-одиночка, "Мать-героиня" |
| **Способ оплаты** | Cashback 2% при оплате через Kaspi Pay |
| **Раннее платёж** | Оплата до 5-го числа — скидка 5% |
| **Месяц дня рождения ребёнка** | В месяц ДР — −10% |
| **Период дат** | 8 марта, Наурыз, НГ — сезонные акции |
| **Первый платёж** | Первый месяц — −50% (акция привлечения) |
| **Композитные AND/OR** | `any_of: [birthday_month, payment_method=kaspi]` |

**Таргетинг:**
- `all` — все активные дети
- `groups` — только указанные группы
- `children` — конкретные дети
- `tariff_types` — только к определённым тарифам
- `age_range` — диапазон возрастов

**Лимиты:** `max_uses_per_child`, `total_max_uses` — защита от злоупотребления.

**Приоритеты:** `priority` + `stackable`. Если `stackable=false` — применяется только одна (с максимальным приоритетом).

### Flow — Создание скидки

1. Admin Web: админ создаёт `custom_discount` (`status='draft'`) → заполняет:
   - Название и описание (ru + kz)
   - Тип (percentage / fixed_amount) и размер
   - Условия (визуальный конструктор → JSONB)
   - Таргет
   - Период (`valid_from`, `valid_until`)
   - Лимиты
   - Текст push-уведомления (опционально)
2. Админ нажимает "Активировать" → `status='active'`.
3. Если `notify_on_activation=true` → enqueue `discount:notify` → worker отправляет push всем родителям таргетных детей с локализованным текстом.
4. При каждой генерации invoice (cron 1-го числа или при разовых начислениях) — `BillingModule` прогоняет активные скидки через `DiscountEngine.evaluate(invoice, discount)`:
   - Проверка условий (composable AND/OR) против `{child, invoice, payment_context}`
   - Проверка таргета и лимитов
   - Если матч → negative `invoice_line_items` + `INSERT custom_discount_applications` + `UPDATE used_count`
5. При истечении `valid_until` → daily cron `discount:expire` (`0 3 * * *` Asia/Almaty) → `status='expired'`. Ручной триггер: `POST /saas/billing/discount-expire-run`.
6. Статус-машина: `draft → active → paused ↔ active | cancelled; active/paused → expired` (по cron). Пауза: `status='paused'` — скидка не применяется при генерации инвойсов. Возобновление: `POST /admin/custom-discounts/:id/resume` → `status='active'` (только из `paused`). Отмена: `status='cancelled'` (финал, не обратим). Скидки применяются только к НОВЫМ инвойсам после активации (backfill не производится).

### Где появляется

- **Admin Web:** список скидок (активные/истёкшие), конструктор условий, превью ("42 ребёнка попадут"), статистика применения.
- **Parent App:** push при активации; в invoice — отдельная строка "Скидка: {название} −{amount}".
- **Staff App:** не видит скидки напрямую (это билинг-фича).

## 5. Daily Operations, Attendance and Child Status

## Process

Ежедневная операционная работа по ребенку в течение дня

### Goal

Фиксировать присутствие, активность и статус ребенка в течение дня.

### Actors

- Mentor
- Reception
- Родитель
- Система

### Platforms

- Staff App
- Parent App

### Main Flows

### 5.1 Check-in утром

1. Ребенок приходит в садик.
2. Check-In
    1. Автоматический `check-in` при помощи **Face ID** турникета.
    2. Сотрудник фиксирует/поправляет`check-in` механически, через Staff App.
3. Система сохраняет время прихода.
4. Система обновляет статус ребенка на `В садике`.
5. Приходит уведомление в Parent App, об обновлении статуса.
6. Родитель видит обновленный статус ребенка (в Разделе События: Проходы) и запись в `timeline`.

### 5.2 Переход между активностями

1. В течение дня группа переходит между активностями.
2. Mentor обновляет текущее событие группы.
    1. Предлагается быстрая кнопка `Следующее событие` с уже готовым событием *(отталкиваясь от расписания)*
    2. Если `Следующее событие` идет не из расписания, то Mentor может выбрать событие из предложенных или создать новое событие
3. Система фиксирует новое событие дня.
    1. Время перехода
    2. Подробности события
4. Родитель видит актуальный статус группы или ребенка.

### 5.3 Внутридневной статус ребенка

1. Сотрудник отмечает внутренний статус или событие по ребенку.
2. Система обновляет `timeline` ребенка.
3. Родитель видит актуализированную историю дня.

### 5.4 Check-out вечером

1. Фиксирование `check-out`.
    1. Обычно `check-out` выполняется по простому Face ID (ребенка) на проходе
    2. Родитель фиксирует по своему Face ID (либо QR) для того, чтобы система могла записывать кто именно забрал ребенка
        1. Доверенное лицо *(OTP)*
        2. Няня *(Свой Mobile App QR или Face ID)*
        3. Второстепенный Родитель *(Свой Mobile App QR или Face ID)*
        4. Primary Guardian *(Свой Mobile App QR или Face ID)*
    3. При необходимости Mentor редактирует данные о проходе
        1. Может менять того - кто забрал, во сколько забрал и т.д.
2. Система сохраняет время выхода и человека, кто забрал.
3. Система обновляет статус ребенка на `Ушел домой`.
4. Родитель видит запись в `timeline` и факт завершения дня.

### 5.5 Корректировка времени

1. Сотрудник открывает историю посещаемости ребенка.
2. Сотрудник корректирует время входа или выхода.
3. Система сохраняет исправленное значение.
4. Родитель видит обновленное время в `timeline`.

### Result

- Все ключевые события дня отражены в системе и видны родителю.

### Дополнения (архитектурные)

- **Face ID стек:** собственный Python-сервис `shyraq-face-service` на FastAPI + InsightFace (`buffalo_l`) + **Silent-Face-Anti-Spoofing (MiniFASNet)** для liveness. Хранение embedding'ов — Qdrant (рекомендую) или pgvector. Камера у входа — с экраном для приветствия.
- **Video enrollment (обязательно):** для каждого subject'а (ребёнок/гвардиан/staff) делается **короткое видео 5–10 сек**, из которого face-service извлекает 5–10 кадров и генерирует множество embedding'ов в `face_embeddings`. Один `face_profile` = N embeddings (robust к позам и освещению). Почему не одно фото: дети крутятся, одна поза даёт много ложных промахов.
- **Anti-spoofing passive:** Silent-Face-Anti-Spoofing работает ПЕРЕД identification. Если `spoof_score < 50` — событие отклоняется (`status='rejected_spoof'`), экран показывает "Покажите лицо, а не фото". Ловит: печатные фото, экраны телефонов. Active liveness (моргнуть) — вне MVP.
- **Check-in поток:** камера детектит → face-service `identify` (anti-spoof → match) → если `confidence ≥ 85%` И `spoof_score ≥ 50` → событие в NestJS → дедуп по Redis `face:seen:{external_id}:{date}` TTL 600s → `attendance_events` (method=`face_id`) + `timeline_entries` + WS + push + приветствие.
- **Fallback сценарии:**
  - `confidence < 85%` → экран: "Не узнали, используйте QR или пройдите к Reception".
  - `spoof_score < 50` → экран: "Покажите лицо, а не фото" (логируется как возможный fraud).
  - **Identity QR** (fallback #2): родитель/сотрудник показывает свой QR → Reception сканирует → `attendance_events (method='qr')` с `pickup_user_id=scanned_user.id`.
  - Reception fix manually через Staff App (`method='manual'`).
- **Check-out с подтверждением забирающего:** лицо ребёнка + лицо забирающего (Primary Guardian / Secondary / Nanny / staff) ЛИБО QR забирающего. `attendance_events.pickup_user_id` указывает фактически забравшего.
- **Биометрия — особая категория ПДн:** хранится только в РК, доступ к face-service через VPN кластера. Требует явного `consent` родителя (см. этап 2).

## Process

Подача и обработка заявок от родителей

### Goal

Организовать управляемый поток заявок от родителей в операционный контур садика.

### Request Types

- **`day_off`** — Выходные дни: родитель просит **оставить ребёнка В САДУ в выходной день** (суббота или воскресенье). Хранится как `details.weekend_dates: DATE[]` — 1 или 2 даты, каждая обязательно суббота или воскресенье. **Не date range.** Обе даты должны быть в одной календарной неделе.
- **`vacation`** — Отпуск: родитель **забирает ребёнка ИЗ САДА на отпуск** (ребёнок не ходит в сад). Хранится как `date_from..date_to` (inclusive range). Оба поля обязательны, `date_from ≥ today`.
- **`late_pickup`** — Поздний забор: одна дата + `expected_time` HH:MM. Счёт (`invoice`) создаётся в B13 (TODO-маркер в B12).
- **`open_request`** — Открытое обращение: subject + message + attachments к admin/mentor/specialist.

### Main Flows

### 6.1 Заявка на доверенное лицо

1. Родитель создает заявку на доверенное лицо в `Parent App`.
2. Родитель вводит Имя и телефон доверенного лица.
3. Родитель подтверждает отправку *(OTP)*.
4. Система создает заявку типа `Доверенное лицо`.
5. Mentor и Admin получают заявку в разделе `Заявки`.
6. Дальше процесс продолжается в `Pickup / Trusted Person / OTP Handoff`.

### 6.2 Заявка на выходные дни (`day_off`)

> **Семантика:** ребёнок **ОСТАЁТСЯ В САДУ** в выходной день (суббота/воскресенье) — родитель просит обеспечить присмотр в нерабочее время садика. Не путать с `vacation` (когда ребёнок не ходит). Wire-поле: `weekend_dates: ["YYYY-MM-DD", ...]` — 1 или 2 даты, каждая суббота или воскресенье. **Не date range.**

1. Родитель создает заявку на выходные дни.
2. Родитель указывает 1 или 2 даты *(суббота и/или воскресенье одной недели)* и комментарий.
3. Родитель подтверждает заявку.
4. Система создает заявку типа `day_off` со статусом `pending`.
5. Mentor видит заявку своей группы.
6. Mentor подтверждает, что заявка принята к сведению (`accept`).
7. Система меняет статус заявки на `accepted`.
8. Родитель видит обновленный статус заявки.
9. Администратор при необходимости видит общую сводку. (Маркер: `// TODO(B22): include accepted day_off weekend_dates in expected attendance`.)

### 6.3 Заявка на поздний забор

1. Родитель создает заявку на поздний забор.
2. Родитель указывает планируемое время забора.
3. Родитель переходит к оплате.
4. После подтверждения оплаты заявка становится активной.
5. Система создает заявку типа `Поздний забор`.
6. Mentor и Admin видят заявку в `Requests`.
7. Сотрудник отмечает заявку как принятую.
8. Система меняет статус заявки на `Принято`.
9. Родитель видит обновленный статус заявки.

### 6.4 Заявка на отпуск (`vacation`)

> **Семантика:** ребёнок **ЗАБИРАЕТСЯ ИЗ САДА на отпуск** (ребёнок не ходит в сад в указанный период). Не путать с `day_off` (когда ребёнок приходит в сад в выходной). Wire-поля: `date_from` + `date_to` (inclusive range), `date_from ≥ today`.

1. Родитель создает заявку на отпускные дни.
2. Родитель указывает `date_from` и `date_to` (inclusive) и комментарий.
3. Родитель подтверждает заявку.
4. Система создает заявку типа `vacation` со статусом `pending`.
5. Mentor видит заявку своей группы.
6. Mentor подтверждает, что заявка принята к сведению (`accept`).
7. Система меняет статус заявки на `accepted`.
8. Родитель видит обновленный статус заявки.
9. Администратор при необходимости видит общую сводку отсутствий. (Маркер: `// TODO(B22): batch-mark child_daily_status='on_vacation' on accept(vacation)`.)

### 6.5 Открытая заявка

1. Родитель создает открытую заявку.
2. Родитель указывает обращение *(Название и Описание)*.
3. Родитель указывает получателя.
    1. Администрация дет. сада
    2. Воспитатель
    3. Специалист *(Указывают конкретного специалиста)*
4. Родитель подтверждает заявку.
5. Система создает заявку типа `Открытая заявка`.
6. Получатель видит заявку.
7. Получатель подтверждает, что заявка принята к сведению.
8. Система меняет статус заявки на `Принято`.
9. Родитель видит обновленный статус заявки.

### State Machine (все типы заявок)

```
pending → accepted   (staff/admin принимает — conditional UPDATE WHERE status='pending')
pending → rejected   (staff/admin отклоняет — conditional UPDATE WHERE status='pending')
pending → cancelled  (родитель отменяет — conditional UPDATE WHERE status='pending')
```

Все три перехода — terminal. Реверс запрещён. Race-guard: `UPDATE … WHERE id=? AND status='pending' RETURNING *` → 0 строк = 409 `parent_request_already_processed`.

### Thread (двусторонний)

Таблица `parent_request_messages`. Как родитель, так и staff/admin могут писать в тред. `author_user_id` (родитель) XOR `author_staff_id` (сотрудник) — ровно одно заполнено (CHECK-constraint). Endpoint'ы: `POST /:id/messages` (обе стороны), `GET /:id/messages?cursor=` (cursor-paged).

### Wire-contract (snake_case)

DTOs на уровне контроллера используют **snake_case** wire-keys (`child_id`, `weekend_dates`, `expected_time`, `is_one_time`, `create_pickup_request`, `recipient_type`, `recipient_staff_id`, `review_note`). Service/domain — camelCase TS.

### Result

- Родительская заявка создана, обработана и получила понятный статус.
- Тред позволяет двустороннюю коммуникацию parent ↔ staff до и после обработки.

## Process

Передача ребенка доверенному лицу через OTP

### Goal

Безопасно подтвердить право забора ребенка не самим родителем.

### Actors

- Родитель
- Доверенное лицо
- Mentor
- Reception
- Система

### Platforms

- Parent App
- Staff App

### Main Flow

1. Родитель заранее добавляет доверенное лицо.
2. Система хранит доверенное лицо в истории заявок ребенка (`trusted_people`).
3. Родитель опционально создаёт `pickup_request` через Parent App (`POST /parent/children/:id/pickup-requests`).
4. В день забора доверенное лицо приходит за ребенком.
5. Mentor или Reception открывает заявку доверенного лица в `Staff App` (или создаёт новый `pickup_request` напрямую через `POST /staff/pickup-requests` если родительской заявки нет).
6. Сотрудник нажимает «Отправить OTP» (`POST /staff/pickup-requests/:id/send-otp`).
7. Система генерирует 6-digit код, пишет `otp:pickup:{requestId}` в Redis TTL 1800с, отправляет SMS на телефон доверенного лица.
8. Доверенное лицо диктует код сотруднику.
9. Сотрудник вводит OTP (`POST /staff/pickup-requests/:id/validate-otp`).
10. Если OTP верный, система атомарно: обновляет `pickup_requests.status='validated'`, создаёт `attendance_events (method='otp_pickup')` + `timeline_entries` + upsert `child_daily_status`, обновляет `attendance_event_id`.
11. Сотрудник завершает передачу ребенка.
12. Родитель получает push `pickup.validated` (recipients: все approved guardians + requester).
13. Запись появляется в `timeline` ребенка.

### Alternative Flows

- **Неверный OTP:** `attempts` инкрементируется; при 3 неверных — `otp:locked:{phone}` TTL 900с, ответ 429 `otp_locked`. Сотрудник не завершает выдачу.
- **OTP истёк:** ключ в Redis отсутствует → 410 `otp_expired`. Сотрудник повторно вызывает `send-otp`.
- **Отмена:** `POST /staff/pickup-requests/:id/cancel` → `status='cancelled'`, DEL Redis ключ.
- **Concurrent validate-otp:** advisory lock `pg_advisory_xact_lock(hashtext('pickup:validate:'||requestId))` через port-метод (паттерн B10 `IdentityQrRepository.acquireUserAdvisoryLock`) — конкурентные validate сериализуются; второй увидит `status='validated'` и получит 409 `pickup_request_already_validated`.

### Separate Simple Flow

### Ребенка забирает сам родитель

1. Сотрудник фиксирует стандартный `check-out` (`POST /staff/attendance/check-out`).
2. Система завершает день без OTP-сценария.
3. Родитель видит факт выхода ребенка.

### Result

- Передача ребенка либо подтверждена безопасным способом, либо отклонена.

### Дополнения (архитектурные)

- **Два режима доверенных лиц:**
  - **Одноразовые** (`trusted_people.is_one_time=true`): родитель добавляет "на один раз" → после успешного pickup `used_at=NOW()` + `is_active=false`. Запись остаётся в истории.
  - **Постоянные (whitelist)** (`is_one_time=false`): переиспользуемый список. Родитель управляет через Parent App (add/revoke).
- **Staff-created pickup_request:** mentor/admin может создать `pickup_request` напрямую без предварительной родительской заявки (`POST /staff/pickup-requests`). Snapshot полей — из whitelist (`trusted_person_id`) или ad-hoc (inline name+phone).
- **OTP инфраструктура:** `OtpStorePort` + `RedisOtpStore` переиспользуются из AuthModule. Namespace `otp:pickup:{requestId}` (не `otp:login:{phone}`). Rate-limit `rate:otp:{phone}` shared с `/auth/otp/request` — один phone = одна квота. OTP TTL = 1800с (= `expires_at - created_at`).
- **Attendance integration:** `AttendanceService.checkOut` принимает опциональный `pickupRequestId` и `method='otp_pickup'`; при передаче validated `pickupRequestId` пропускает стандартную проверку `findApprovedActivePickupGuardian` (доверенное лицо не guardian).
- **Notifications:** `pickup.otp_sent` — SMS на телефон доверенного напрямую через `SmsPort`; запись в `notification_outbox` для аудита и push-уведомления инициатору. `pickup.validated` — guardians + requester получают push.
- **OTP vs QR как альтернативы:**
  - **SMS OTP** (текущий флоу, B11): `pickup_request` → SMS на телефон доверенного → OTP диктуется сотруднику.
  - **QR (альтернатива):** родитель генерирует одноразовый QR в Parent App → пересылает доверенному → доверенный показывает на входе → Reception сканирует через Staff App → валидация → check-out. Удобно, если у доверенного плохая связь / нет SMS.
  - (QR для доверенных — вне MVP, но архитектурно заложен через тот же `user_qr_tokens` с расширением `purpose`.)
- **Идентификация забирающего через Face ID:** если у trusted-person есть face_profile (был зарегистрирован ранее) — автоматическое распознавание на турникете заменяет OTP.
- **Identity QR родителя/Primary Guardian** — основной fallback-механизм идентификации, если Face ID не сработает.

- **Staff (Mentor/Specialist)** — read-only: просмотр меню своей группы или общесадикового.
- **Родитель** — read-only: просмотр меню по дате/неделе/месяцу через кабинет ребёнка.

**Main Flow:**

1. Администратор открывает раздел меню в `Admin Web`.
2. Администратор создаёт `meal_plan` на нужную дату (опционально — для конкретной группы).
3. Администратор добавляет блюда по типам (`breakfast`, `lunch` и т.д.) с локализованными названиями.
4. Система сохраняет меню (`source='manual'`, `is_published=true`).
5. Родители видят актуальное меню в `Parent App` через `GET /parent/children/:id/menu?date_from=...&date_to=...`.

**Идемпотентность создания:** уникальный индекс `(kindergarten_id, date, group_id)` (где `group_id IS NULL` означает общее меню садика) предотвращает дублирование. При конфликте → 409 `meal_plan_already_exists`.

### 9.3 Обновление расписания

**Структура данных:**

- `schedule_templates` — именованный шаблон недельного расписания для группы (`group_id`). Шаблон имеет `valid_from`, `valid_until` (опционально) и `recurrence='weekly'`. Один активный шаблон = недельный паттерн для группы.
- `schedule_template_slots` — слоты шаблона: `day_of_week` (mon/tue/wed/thu/fri/sat/sun) × `start_time`/`end_time` × `activity_name`. Уникальность: `(template_id, day_of_week, start_time)` — нельзя создать два одинаковых слота в одно и то же время.
- `activity_events` — конкретные dated-события (с `starts_at` timestamptz). Создаются проекцией из template-slots на конкретную дату при авто-копировании или вручную staff'ом.
- `schedule_week_snapshots` — флаг-запись о наличии расписания на конкретную неделю для группы `(group_id, week_start_date)`.

**State machine `activity_event.status`:**

- `scheduled` — создано, ожидает начала (исходное состояние).
- `in_progress` — mentor нажал "Начать" в Staff App. Обновляет `groups.current_location_id` → WS broadcast.
- `completed` — событие завершено.
- `cancelled` — событие отменено (например, болезнь группы, выезд).

Допустимые переходы: `scheduled → in_progress → completed`, `scheduled → cancelled`, `in_progress → cancelled`.

**Акторы и права:**

- **Admin** — полный CRUD шаблонов (`schedule_templates`, `schedule_template_slots`); просмотр и ручное создание `activity_events`; ручной запуск copy-week.
- **Staff (Mentor)** — read-only расписание своей группы; переключение статусов `activity_events` (`start`/`complete`/`cancel`) и создание внеплановых событий.
- **Родитель** — read-only расписание группы своего ребёнка через `GET /parent/children/:id/schedule`.

**Main Flow:**

1. Администратор создаёт `schedule_template` для группы с `valid_from`.
2. Администратор добавляет `schedule_template_slots` (например, Mon 09:00–09:45 "Утренний круг", Mon 10:00–11:00 "ИЗО").
3. Cron (воскресенье 23:00 Asia/Almaty) или ручной `POST /admin/schedule/week-snapshots/copy` создаёт `schedule_week_snapshots` + `activity_events` на следующую неделю из актуального шаблона.
4. В течение дня Mentor в Staff App видит события своей группы, переводит их через статусы.
5. Родители видят актуальное расписание группы ребёнка в `Parent App`.

**Видимость для родителя:** родитель видит `activity_events` группы, к которой принадлежит его ребёнок, на запрошенный диапазон дат. Показываются все статусы (scheduled/in_progress/completed/cancelled). Фильтрация по дате передаётся через query-параметры `date_from` / `date_to`.

### 9.4 Публикация нового Qundylyq

1. Администратор обновляет
    1. тему месяца, 
    2. описание 
    3. связанный контент.
2. Система сохраняет новый `Qundylyq`.
3. Родители видят обновленный блок `Qundylyq`.

### 9.5 Поздравление с днем рождения

1. Система определяет ближайший день рождения.
2. Администратор открывает список дней рождения.
3. Администратор публикует запланированный поздравительный пост.
    1. Публикуется в день рождения ребенка
    2. Автогенерация и публикация при отсутствии запланированного поста
4. Родители видят поздравление в ленте.

### 9.6 Stories группы

1. Mentor снимает фото или видео и добавляет описание в `Staff App`.
2. Mentor публикует `stories` для активной группы.
3. Система сохраняет `stories` с сроком жизни 24 часа.
4. Родители этой группы видят `stories` в своей ленте.

### Result

- Родители получают актуальный контент по ребенку, группе и саду.

### Дополнения (архитектурные)

- **Меню — структурированно, не как новость:** `meal_plans` (per date, optional per group) + `meal_items` (breakfast / snack_am / lunch / snack_pm / dinner) с `dish_name` и `description` в JSONB `{ru, kz}`, `allergens[]`, `calories`, `photo_url`. Просмотр на день/неделю/месяц — запрос по диапазону дат.
- **Расписание — структурированно:** `schedule_templates` + `schedule_template_slots` (weekly pattern) → проекция в `activity_events` по датам. `schedule_week_snapshots` — флаг наличия недели.
- **Auto-copy следующей недели** (cron каждое воскресенье 23:00):
  - **Меню:** если на следующую ПН–ПТ нет `meal_plans` → копируем с текущей недели (сдвиг +7 дней, `source='auto_copied_from_previous_week'`, `copied_from` ссылка).
  - **Расписание:** если нет `schedule_week_snapshots` на следующую неделю → копируем `schedule_template_slots` и создаём `activity_events`.
  - Админ может править после авто-копирования — следующее копирование учтёт новое.
- **Локализация контента:** `content_posts.title_i18n`, `body_i18n`, `meal_items.dish_name` — JSONB `{ru, kz}`. Резолв по `users.locale` с fallback на RU.
- **Планирование публикации:** `content_posts.scheduled_for` + статус (`draft`/`scheduled`/`published`). Cron публикатор проверяет каждые 5 минут.
- **Stories TTL:** `group_stories.expires_at = created_at + 24h`. Cron `stories:cleanup` раз в час удаляет истёкшие (+ S3 lifecycle rule 48h на страховку).

## 12. Lifecycle of Child in the System

## Process

Жизненный цикл ребенка в системе

### Goal

Управлять статусом ребенка от создания карточки до архива без потери истории.

### Actors

- Администратор
- Mentor
- Родитель
- Система

### Platforms

- Admin Web
- Staff App
- Parent App

### Main Flows

### 12.1 Первичное создание ребенка

1. Администратор создает карточку ребенка в `Admin Web`.
2. Система сохраняет ребенка в статусе `Создана карточка`.
3. После привязки родителя и первой оплаты ребенок становится активным.

### 12.2 Перевод в другую группу

1. **Триггер:** Администратор открывает карточку ребенка и вызывает `POST /admin/children/:id/transfer-group {to_group_id, reason?}`.
2. Администратор изменяет группу и при необходимости назначенного Mentor.
3. Система: `UPDATE children.current_group_id` + INSERT в `child_group_history` (`from_group_id`, `to_group_id`, `transferred_by=staff_member_id`, `reason`) — audit-trail.
4. Mentor'ы старой и новой группы получают уведомление `child.transferred` (через `NotificationPort` skeleton в B4 → real BullMQ + FCM/APNS + WS в B9). Primary guardian тоже получает уведомление.
5. Новый Mentor получает ребенка в своей группе в `Staff App`.
6. Родитель видит новую группу ребенка в `Parent App`.
7. Система продолжает вести историю ребенка без потери прошлых данных (`child_group_history` хранится навсегда).

### 12.3 Смена Ментора/Специалиста группы

1. Администратор открывает карточку группы.
2. Администратор изменяет назначенного Ментора/Специалиста.
3. Система сохраняет новую привязку.
4. Новый Mentor/Специалист получает ребенка в своей группе в `Staff App`.
5. Родитель видит нового Ментора/Специалиста ребенка в `Parent App`.
6. Система продолжает вести историю группы без потери прошлых данных.

### 12.4 Изменение состава законных представителей

1. Администратор обновляет данные родителей или связи с аккаунтами.
2. Система сохраняет новую конфигурацию доступов.
3. Актуальный родитель получает доступ к ребенку.
4. Неактуальный доступ при необходимости отключается.

**State-transitions для guardian-записи (B4):**
- `pending_approval → approved` — primary одобряет.
- `pending_approval → rejected` — primary отклоняет (конечное состояние).
- `approved → revoked` — primary или admin отзывает (`revoked_at`, `revoked_by`).
- `rejected` — конечное.

**Approval-flow (B4):**
1. Admin вызывает `POST /admin/children/:id/guardians` с `{user_phone OR user_id, role, can_pickup, has_approval_rights?}` — find-or-create user by phone (паттерн B2 D10 L3), `child_guardians.status='pending_approval'`.
2. Primary guardian получает уведомление `guardian.pending_approval` (через `NotificationPort` skeleton).
3. Primary вызывает `POST /parent/approvals/:guardianId/approve` — система проставляет `status='approved'`, `approved_by=me`, `approved_at=NOW()` и **seed'ит** `permissions` дефолтами из роли (см. `endpoints.md §4.13`). Опционально включает `has_approval_rights` (инвариант: ≤2 на ребёнка). Notify `guardian.approved` admin'у и guardian'у.
4. Альтернативно `POST /parent/approvals/:guardianId/reject` (notify `guardian.rejected`) или `POST /parent/approvals/:guardianId/revoke` (notify `guardian.revoked`).

**Permissions edit (Primary, OTP-flow):**
1. Primary открывает экран «Права» для secondary/nanny. `GET /parent/children/:id/guardians/:guardianId/permissions` — эффективные права (defaults role + overrides).
2. Меняет toggle'ы → `POST .../permissions/otp-request {permissions}` — сервер считает `payload_hash=SHA256(stable_json(permissions))`, шлёт SMS на phone primary'а, пишет Redis `otp:guardian-permissions:{primary_user_id}:{guardianId}` Hash `{code, attempts, payload_hash}` TTL 300s.
3. `PATCH .../permissions {permissions, otp_code}` — сервер проверяет `code` И `payload_hash == SHA256(stable_json(body.permissions))`; при совпадении — `UPDATE child_guardians SET permissions = permissions || $patch (JSONB merge)`, `permissions_updated_by=me`, DEL Redis-ключ, broadcast `guardian.permissions_updated` затронутому guardian'у.
4. При 3 неверных кодах — `otp:locked:{phone}` TTL 900s (общий lockout pattern с auth OTP, B1).

**Locked-keys (нельзя через permissions-patch):** `has_approval_rights`, `prepayment`, `trusted_people_manage`, `approvals` — управляются отдельным flow (`PATCH /parent/approvals/:gid/rights` для `has_approval_rights`, остальные defaults role).

**Инварианты:**
- ≤2 guardian'а с `has_approval_rights=true` per child (DB-trigger `trg_child_guardians_approval_max2`).
- `revoked_at IS NOT NULL ⇒ status='revoked'`.
- Nanny не получает primary/secondary defaults для `view_payments`, `pay_invoices`, `view_diagnostics`, `create_requests` — только `view_timeline` и pickup-связанные права.
- **No limit на общее число nanny-записей per child** (только ≤2 approval_rights enforced).

### 12.5 Завершение посещения садика

1. Администратор переводит ребенка в архив.
2. Система закрывает активные операционные процессы ребенка.
3. Ребенок исчезает из активных рабочих списков в `Staff App`.
4. Кабинет ребенка в `Parent App` переводится в архивный режим или ограниченный доступ.
5. Система сохраняет историю посещения, оплат, заявок и прогресса.

### Result

- История ребенка сохраняется на всем жизненном цикле, а активный статус меняется управляемо.

### Дополнения (архитектурные)

- **Архивирование → остановка биллинга:** при `UPDATE children.status='archived'` система проставляет `archived_at`, закрывает активные `activity_events`, обновляет `tariff_assignments.valid_until = today`. Cron больше не генерирует invoice для этого ребёнка.
- **Pro-rata refund при архивировании посреди месяца:** enqueue `billing:pro-rata` → worker считает billable-дни до `archived_at` (с учётом `kindergarten_holidays.is_billable=false`) → создаёт `refunds (status='pending')` для подтверждения админом ЛИБО negative `invoice_line_items` на следующий invoice.
- **Реактивация:** возврат в `status='active'`, открытие новой `tariff_assignments` (`valid_from=today`), разблокировка Parent App.
- **История доступов:** изменения в `child_guardians` (отзыв, добавление новых) логируются через `updated_at` + поля `revoked_by`, `approved_by`, `approved_at`, `revoked_at`. Полноценный audit_log — этап 2.
- **Retention:** данные не удаляются даже после архива — только ограничивается доступ родителя. Удаление по запросу субъекта ПДн (ZRK 94-V) — отдельный процесс, запланирован на этап 2 (`data_export_requests`, `purge` endpoint).

## 13. Identity QR System (сквозной механизм)

## Process

Выдача и использование персонального QR каждого пользователя как fallback-идентификатор.

### Goal

Обеспечить **второй контур идентификации** на случай сбоя Face ID и универсальный идентификатор юзера для разных действий.

### Actors

- Родитель / Сотрудник / Доверенное лицо (в будущем)
- Reception / Mentor
- Система

### Platforms

- Parent App (свой QR)
- Staff App (свой QR + сканирование)

### Main Flow

1. При первом `GET /users/me/qr` → сервер lazy-генерирует `user_qr_tokens (purpose='identity', TTL=24h)`. Токен = 32-char opaque hex (`crypto.randomBytes(16).toString('hex')`). SHA-256 hash сохраняется в `user_qr_tokens.token_hash`; plaintext только в Redis `qr:token:{plaintext}` TTL=24h.
2. Redis кэш: `qr:token:{plaintext}` = `user_id`, TTL = expires_at - now — для O(1) валидации.
3. Приложение (Parent/Staff) получает `{token, issued_at, expires_at}` и рендерит QR-картинку нативно (сервер не генерирует SVG и не возвращает `qr_svg_url`). На повторном GET — если активный токен ещё свеж (`expires_at - now > 1 час`) → сервер возвращает тот же plaintext (из `qr:user:{userId}:identity` Redis-ключа); если активного нет или до `expires_at` осталось <1 часа → авто-ревокирует старый и создаёт новый в одной транзакции (revoke + insert + Redis sync обоих ключей). Ручного refresh-endpoint'а нет.
4. Reception/Mentor в Staff App сканирует QR → `POST /staff/qr/scan` → Redis GET → вернуть user info + разрешённые действия.
5. Каждый скан логируется: `user_qr_tokens.last_scanned_at`.

### Use Cases

| Сценарий | Кто сканирует | Действие |
|---|---|---|
| **Fallback check-in/check-out** | Reception | Face ID не сработал → сканирует QR ребёнка (из Parent App родителя, если ребёнок слишком мал для своего QR) |
| **Подтверждение забора родителем** | Reception | Родитель показывает свой QR → `attendance_events.method='qr'`, `pickup_user_id=user.id` |
| **Несколько детей в одном саду** | Reception / Mentor | Если у родителя несколько детей в том же саду — `linkedChildren` массив со всеми; Staff App отображает все карточки и check-out по одному (B8 per-child API). |
| **Cross-kg родитель** | Reception / Mentor | Если у родителя дети в разных садиках — `linkedChildren` содержит только детей из сада сканирующего сотрудника. Идентификация (имя/телефон/avatar) показывается всегда (token cross-tenant), но дети из другого сада никогда не утекают. |
| **Staff авторизация у турникета** | Турникет с камерой+сканером | Сотрудник без face_profile использует QR |
| **Доверенное лицо (future)** | Reception | QR вместо SMS OTP |

### Security

- TTL 24h; автоматический серверный refresh: `GET /users/me/qr` переиспользует активный токен, если `expires_at - now > 1ч`, и mintsит fresh иначе. Сервер хранит активный plaintext в `qr:user:{userId}:identity` (TTL = remaining lifetime), парном к `qr:token:{plaintext} → user_id`, чтобы recover plaintext на повторном GET без повторного minting. Каждое issueOrRefresh начинается с `pg_advisory_xact_lock(hashtext('qr:identity:'||user_id))` — сериализует одновременные `GET /users/me/qr`: первый mintsит, остальные читают cache и возвращают тот же plaintext, не упираясь в `idx_user_qr_tokens_one_active`.
- **Admin revoke-all:** `POST /admin/qr/revoke-all/:userId` — bulk-revoke всех активных токенов пользователя: DB `revoked_at = NOW()` для всех активных строк + `DEL qr:user:{userId}:identity` (next user GET сразу mintsит fresh, а не возвращает кэшированный плэйнтекст к уже revoked-строке). Plaintext-keyed `qr:token:{plaintext}` НЕ удаляется (admin имеет только хэш) — следующий скан попадает в DB-recheck и возвращает 410 `qr_token_revoked`; cache-TTL ≤24h задаёт верхнюю границу stale-cache exposure. Доступ: только если target user — активный `staff_member` в kg сканирующего админа ИЛИ approved+non-revoked `child_guardian` ребёнка из kg сканирующего админа. Иначе 403 `user_no_relationship_to_kindergarten`. Если userId не существует — 404 `user_not_found`. Возвращает `{revoked_count}`. Security-recourse при подозрении на компрометацию. Включён в MVP.
- Rate-limit на `POST /staff/qr/scan`: 60/мин per `device_id` (берётся из заголовка `X-Device-Id` + проверяется через `hasActiveSessionForDevice(userId, deviceId)` против активной `refresh_tokens` сессии вызывающего staff — без этой связки header можно было бы крутить и обходить лимит). Redis: `rl:qr:scan:{device_id}` INCR+EXPIRE 60. `X-Device-Id` персистится на `refresh_tokens.device_id` при OTP-verify / refresh / role-select.
- Логирование всех сканов с IP и device_id (в будущем — `audit_log`).
- **Token cross-tenant; `linkedChildren` per-kg.** `user_qr_tokens.kindergarten_id` nullable — токен идентифицирует пользователя глобально (один QR для родителя с детьми в нескольких садиках), RLS не применяется к таблице. Но в ответе `POST /staff/qr/scan` массив `linkedChildren` фильтруется по `kindergarten_id` сканирующего сотрудника: staff в kg-A, сканирующий родителя с детьми в kg-A и kg-B, видит только kg-A ребёнка(детей); kg-B ребёнок в его ответе никогда не появляется. `allowedActions` тоже считается по guardian-связям только в kg-A (нет `can_pickup` в kg-A → `[]`).

### Result

- У каждого юзера есть актуальный QR.
- Любая идентификация возможна минимум двумя способами (Face ID + QR).

## Открытые риски (вне MVP)

Следующие пункты не входят в MVP, но являются юридическими/коммерческими рисками и должны быть реализованы до платного запуска:

- **Инциденты / травмы** (`incident_reports`) — юридически важный журнал ЧП с подписью родителя.
- **Документы ребёнка** (`child_documents`) — свидетельство о рождении, мед.справка, прививочная карта.
- **Журнал здоровья** (`child_health_log`) — температура, приём лекарств.
- **Audit log** — следы изменений персональных данных и денежных операций.
- **Consents** — явные согласия родителя на обработку ПДн, фото, CCTV, **Face ID (биометрия — особая категория)**. Требование закона РК "О персональных данных".
- **Data export / delete by request** — право субъекта ПДн на выгрузку и удаление.
- **Чат parent ↔ mentor** — конкурентный недостаток; оценивается после pilot.
